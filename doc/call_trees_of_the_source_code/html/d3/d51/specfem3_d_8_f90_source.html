263 ! Reference frame - convention:
264 ! ----------------------------
265 !
266 ! The code uses the following convention for the reference frame:
267 !
268 ! - X axis is East
269 ! - Y axis is North
270 ! - Z axis is up
271 !
272 ! Note that this convention is different from both the Aki-Richards convention
273 ! and the Harvard CMT convention.
274 !
275 ! Let us recall that the Aki-Richards convention is:
276 !
277 ! - X axis is North
278 ! - Y axis is East
279 ! - Z axis is down
280 !
281 ! and that the Harvard CMT convention is:
282 !
283 ! - X axis is South
284 ! - Y axis is East
285 ! - Z axis is up

363 ! From Dahlen and Tromp (1998):
364 ! ----------------------------
365 !
366 ! Gravity is approximated by solving eq (3.259) without the Phi_E' term
367 ! The ellipsoidal reference model is that of section 14.1
368 ! The transversely isotropic expression for PREM is that of eq (8.190)
369 !
370 ! Formulation in the fluid (acoustic) outer core:
371 ! -----------------------------------------------
372 !
373 ! In case of an acoustic medium, a displacement potential Chi is used
374 ! as in Chaljub and Valette, Geophysical Journal International, vol. 158,
375 ! p. 131-141 (2004) and *NOT* a velocity potential as in Komatitsch and Tromp,
376 ! Geophysical Journal International, vol. 150, p. 303-318 (2002).
377 ! This permits acoustic-elastic coupling based on a non-iterative time scheme.
378 ! Displacement if we ignore gravity is then: u = grad(Chi)
379 ! (In the context of the Cowling approximation displacement is
380 ! u = grad(rho * Chi) / rho, *not* u = grad(Chi).)
381 ! Velocity is then: v = grad(Chi_dot) (Chi_dot being the time derivative of Chi)
382 ! and pressure is: p = - rho * Chi_dot_dot (Chi_dot_dot being the time second derivative of Chi).
383 ! The source in an acoustic element is a pressure source.
384 ! The potential in the outer core is called displ_outer_core for simplicity.
385 ! Its first time derivative is called veloc_outer_core.
386 ! Its second time derivative is called accel_outer_core.

399 ! trivia about the programming style adopted here:
400 !
401 ! note 1: for performance reasons, we try to use as much from the stack memory as possible.
402 ! This is done to avoid memory fragmentation and also to optimize performance.
403 ! Stack memory is a place in computer memory where all the variables that are declared
404 ! and initialized **before** runtime are stored. Our static array allocation will use that one.
405 ! All variables declared within our main routine also will be stored on the stack.
406 !
407 ! the heap is the section of computer memory where all the variables created or initialized
408 ! **at** runtime are stored. it is used for dynamic memory allocation.
409 !
410 ! stack is much faster than the heap.
411 !
412 ! when calling a function, additional storage will be allocated for the variables in that function.
413 ! that storage will be allocated in the heap memory segment.
414 !
415 ! most routine calls here will have rather long argument lists, probably because of this performance criteria.
416 ! using modules/common data blocks together with dynamic allocation will put data into heap memory,
417 ! thus it has longer latency to access variables than stack memory variables.
418 !
419 ! however, declaring the static arrays needed in compute_forces_crust_mantle_Dev()
420 ! like e.g. sum_terms, tempx1,... in this main routine and
421 ! passing them along as arguments to the routine makes the code slower.
422 ! it seems that this stack/heap criterion is more complicated,
423 ! and inlining functions is a performance criteria as well.
424 !
425 ! for vectorization, we assume that arrays have contiguous memory allocated. this holds true for most compilers and
426 ! static memory allocation. however, note that dynamically allocated memory could in principle be non-contiguous.
427 !
428 ! another reason why the use of modules is restricted is to make the code thread safe.
429 ! having different threads access the same data structure and modifying it at the same time
430 ! would lead to problems. passing arguments is a way to avoid such complications.
431 !
432 ! note 2: Most of the computation time is spent
433 ! inside the time loop (mainly in the compute_forces_crust_mantle_Dev() routine).
434 ! Any code performance tuning will be most effective in there.
435 !
436 ! note 3: Fortran is a code language that uses column-first ordering for arrays,
437 ! e.g., it stores a(i,j) in this order: a(1,1),a(2,1),a(3,1),...,a(1,2),a(2,2),a(3,2),..
438 ! it is therefore more efficient to have the inner-loop over i, and the outer loop over j
439 !
440 ! note 4: Deville et al. (2002) routines significantly reduce the total number of memory accesses
441 ! required to perform matrix-matrix products at the spectral element level.
442 ! For most compilers and hardware, will result in a significant speedup (> 30% or more, sometimes twice faster).
443 !
444 ! note 5: whenever adding some new code, please make sure to use
445 ! spaces rather than tabs. Tabulators are in principle not allowed in Fortran95.
446 !
